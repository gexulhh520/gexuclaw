# FastAPI + Python 开发规范

这份规范适合你在现有项目里长期使用，目标是让后端开发保持：

- 结构清晰
- 接口统一
- 代码可维护
- 容易排查问题
- 方便和前端协作

建议把它当成项目默认约定。后续不管是你自己写，还是让大模型帮你写，都尽量遵守这套规则。

---

## 1. 技术栈约定

推荐默认约定：

- Python 3.10+
- FastAPI
- pytest 做测试
- ruff / black / isort 做代码规范

如果项目已经有现成方案，优先沿用现有方案，不要为了“更标准”轻易切栈。

---

## 2. 总体原则

### 2.1 先一致，再扩展

所有接口、返回值、错误码、命名方式都要保持一致，避免同一个项目里风格混乱。

### 2.2 能复用就复用

公共的：

- 响应格式
- 异常处理
- 校验逻辑
- 权限逻辑
- 数据库操作
- 日志封装

尽量抽到统一位置，不要在每个接口里重复写。

### 2.3 面向真实维护

代码不只是“跑通”，还要考虑：

- 后续是否好排查
- 是否容易扩展
- 是否方便前端理解
- 是否容易写测试

---

## 3. 目录结构建议

推荐一个清晰、可扩展的结构：

```text
app/
  main.py
  core/
    config.py
    logging.py
    security.py
    exceptions.py
    deps.py
  api/
    v1/
      routers/
        user.py
        auth.py
        order.py
      api.py
  models/
    user.py
    order.py
  schemas/
    user.py
    order.py
    common.py
  services/
    user_service.py
    order_service.py
  crud/
    user.py
    order.py
  db/
    session.py
    base.py
    migrations/
  utils/
    time.py
    ids.py
    response.py
  tests/
    test_user.py
    test_auth.py
```

说明：

- `api/` 只负责路由和请求响应
- `schemas/` 放 Pydantic 数据结构
- `services/` 放业务逻辑
- `crud/` 放数据库访问
- `models/` 放 ORM 模型
- `core/` 放配置、异常、日志、安全相关基础能力

如果你的项目已经有类似结构，不建议大改，保持一致最重要。

## 4. 接口规范

### 4.1 路由命名

建议使用资源化命名：

- `GET /users`
- `POST /users`
- `GET /users/{user_id}`
- `PUT /users/{user_id}`
- `DELETE /users/{user_id}`

避免混乱的动词式接口，除非是明确的动作型业务：

- `POST /auth/login`
- `POST /auth/logout`
- `POST /orders/{id}/submit`

### 4.2 版本管理

建议统一使用版本前缀：

- `/api/v1/...`

以后升级时更稳，不会把老接口直接打坏。

### 4.3 请求方法约定

- `GET`：查询
- `POST`：创建、提交动作、复杂查询
- `PUT`：整体更新
- `PATCH`：部分更新
- `DELETE`：删除

### 4.4 参数风格

- 路径参数用资源 ID
- 查询参数用于筛选、分页、排序
- 请求体用于创建和更新

---

## 5. 统一响应格式

建议所有接口统一返回结构，避免前端处理混乱。

推荐格式：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "xxxx"
}
```

建议约定：

- `code = 0` 表示成功
- 非 `0` 表示失败
- `message` 用于人类可读提示
- `data` 放业务数据
- `request_id` 方便排查日志

如果是分页数据，建议统一结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "page_size": 20
  }
}
```

## 6. Pydantic 规范

### 6.1 Schema 分层

建议把请求和响应分开：

- `UserCreate`
- `UserUpdate`
- `UserDetail`
- `UserListItem`

不要一个模型到处混用。

### 6.2 类型完整

字段尽量写清楚类型和默认值，避免隐式行为。

### 6.3 校验前置

能在 Pydantic 层做的校验，尽量不要拖到业务层。

例如：

- 字符串长度
- 枚举值
- 必填字段
- 格式校验

---

## 7. 业务层规范

### 7.1 路由只做薄层

路由函数不要塞太多业务逻辑，推荐结构：

1. 接收参数
2. 调用 service
3. 返回统一响应

### 7.2 service 负责业务编排

业务流程、权限判断、数据组合、状态流转，尽量放在 service。

### 7.3 crud 只负责数据访问

CRUD 层只处理数据库读写，不要夹杂复杂业务判断。

---

## 8. 异常处理规范

### 8.1 统一异常出口

建议把常见异常统一处理：

- 参数错误
- 资源不存在
- 权限不足
- 数据冲突
- 业务校验失败
- 系统异常

### 8.2 错误信息要可理解

给前端的错误信息要明确，但不要泄露敏感内部细节。

例如：

- 不建议：`sqlalchemy.exc.IntegrityError: duplicate key`
- 建议：`用户名已存在`

### 8.3 区分业务异常和系统异常

- 业务异常：预期内问题，例如重复提交、权限不足
- 系统异常：程序错误、数据库错误、未知异常

---

## 9. 数据库规范

### 9.1 ORM 模型命名

建议：

- 表名用复数或统一风格
- 字段名使用 `snake_case`
- 主键统一 `id`

### 9.2 迁移规范

所有结构变更都走 Alembic 迁移，不要手改数据库后忘记同步代码。

### 9.3 查询优化

注意：

- 避免 N+1 查询
- 列表接口只取必要字段
- 大分页要考虑性能
- 关联查询要明确控制

---

## 10. 认证与权限

### 10.1 认证

建议统一处理登录态：

- JWT
- Session
- OAuth2

选一种主方案，不要混乱。

### 10.2 权限

建议把权限逻辑抽成独立层：

- 是否登录
- 是否有角色
- 是否有资源权限
- 是否允许操作当前对象

### 10.3 不要把权限判断散落各处

权限判断最好集中，方便审计和维护。

---

## 11. 日志规范

### 11.1 必须有结构化日志意识

建议记录：

- 请求路径
- 请求方法
- 用户标识
- request_id
- 耗时
- 错误堆栈

### 11.2 日志不要乱打

避免在业务代码里四处 `print`。

### 11.3 敏感信息脱敏

日志里不要泄露：

- 密码
- token
- 身份证号
- 手机号明文
- 关键隐私数据

---

## 12. 配置管理

建议使用环境变量区分配置：

- 开发环境
- 测试环境
- 生产环境

配置项建议集中管理：

- 数据库地址
- Redis 地址
- JWT 密钥
- 日志级别
- 调试开关

不要把配置散落在代码里。

---

## 13. 测试规范

### 13.1 至少覆盖关键路径

建议优先测：

- 登录
- 权限
- 核心 CRUD
- 异常分支
- 边界条件

### 13.2 单测和接口测试分开理解

- 单测：测试函数和业务逻辑
- 接口测试：测试 API 的输入输出和状态

### 13.3 测试数据要可控

不要依赖“手工环境碰巧存在的数据”。

---

## 14. 代码风格规范

建议统一：

- 使用 `ruff` 做基础检查
- 使用 `black` 格式化
- 使用 `isort` 整理 import
- 函数职责尽量单一
- 命名清晰，避免缩写过度

建议：

- 函数名表达动作
- 类名表达职责
- 变量名表达语义

例如：

- `get_user_detail`
- `create_order`
- `validate_permission`
- `user_service`

---

## 15. 接口文档规范

建议每个接口都明确：

- 接口路径
- 请求方法
- 参数说明
- 返回结构
- 错误码
- 权限要求

FastAPI 本身会生成 OpenAPI 文档，建议把注释和描述写完整，方便前后端协作。

---

## 16. 常见开发约束

### 16.1 不要在路由层写大段业务逻辑

### 16.2 不要在 service 里直接拼大段 SQL，除非项目明确这么做

### 16.3 不要让一个函数同时负责校验、查询、写库、返回

### 16.4 不要让模型层、接口层、业务层彼此耦合太深

### 16.5 不要忽略异常分支和边界条件

---

## 17. 给大模型用的后端提示词模板

如果你希望大模型也遵守这套规范，可以直接用下面这段。

```text

要求：
- 优先遵守当前项目已有目录结构、命名方式、编码风格
- 不要为了完成任务随意引入新依赖
- 路由层保持薄，不要堆业务逻辑
- 业务逻辑放在 service 层
- 数据访问放在 crud / repository 层
- 使用统一的响应格式
- 补全异常处理、参数校验、日志和测试思路
- 保持代码清晰、可维护、易扩展

工程规范：
- Python 3.10+
- FastAPI
- Pydantic
- 使用类型注解
- 避免过度抽象

输出要求：
1. 先分析需求在项目中的位置
2. 再说明实现思路
3. 最后给出具体代码
4. 如果有潜在风险，请明确指出
```

## 18. 最推荐的落地方式

建议你把前后端各自的规范分开管理：

- 前端：Vue 3 + TS + Vite 规范
- 后端：FastAPI + Python 规范

然后再补一份跨端协作约定：

- 接口返回格式统一
- 错误码统一
- 分页结构统一
- 时间格式统一
- 字段命名统一

这样后面你让大模型继续开发时，质量会稳定很多。
