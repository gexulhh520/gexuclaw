正确设计

create_work_context 应该分两种：

1. create_work_context + plan.steps 有值
   = 创建 WorkContext 后立即执行

2. create_work_context + plan=null
   = 只创建 WorkContext，不执行，返回等待/缺少执行器提示

所以 create_work_context 不应该强制要求 plan.steps。

真正需要 plan.steps 的应该只有：

const requiresPlan = [
  "delegate",
  "multi_step_plan",
  "recover_execution",
  "verify_execution",
].includes(decision.decisionType);

而不是包含 create_work_context。

还要修 Prompt，避免 LLM 过多输出 create_work_context

你现在的 LLM 判断“没有文件系统 Agent，所以只创建 WorkContext”是合理的，但如果你的系统其实有浏览/文件 Agent，只是 availableAgents 描述不明确，那 prompt 和 Agent 能力配置也要加强。

在 buildMainDecisionSystemPrompt() 里补一句：

如果用户任务需要执行，但 availableAgents 中没有任何 Agent 能完成该任务，可以输出 decisionType=create_work_context，targetWorkContextUid=null，createWorkContext 填写 title/goal，plan=null，并在 reasoning 中说明缺少合适执行器。

同时把这一句改掉：

delegate、recover_execution、verify_execution、multi_step_plan 必须填写 plan.steps。

保持不含 create_work_context。

如果现在 prompt 里有类似“create_work_context 必填 plan”，也要删掉。

最终应该支持的合法输出

这次日志对应的 MainDecision 应该被视为合法：

{
  "decisionType": "create_work_context",
  "targetWorkContextUid": null,
  "createWorkContext": {
    "title": "查看工作目录文件列表",
    "goal": "探索当前工作目录下的文件和目录结构"
  },
  "primaryRefs": [],
  "secondaryRefs": [],
  "targetAgentUid": null,
  "plan": null,
  "response": null,
  "ambiguity": null,
  "confidence": "medium",
  "reasoning": "用户询问工作目录文件列表，这是新的文件系统探索任务；当前没有具备文件系统能力的 Agent，先创建上下文等待执行器。"
}

后端应该：

1. Validator 通过
2. OrchestrationService 创建 WorkContext
3. 不调用 PlanCompiler
4. main run 返回：已创建上下文，但缺少合适执行 Agent

一句话：create_work_context 是“创建上下文动作”，不是一定要执行子 Agent 的动作。plan.steps 只应该强制给 delegate/multi_step_plan/recover/verify。
