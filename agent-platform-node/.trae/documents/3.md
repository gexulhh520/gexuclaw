改进目标

当前已经完成了：

1. TaskEnvelope 不默认渲染 Original User Message
2. Selected Context 默认隐藏，避免重复
3. artifacts 支持 contentText/contentJson 渲染
4. step dependsOn 可以自动合并上一步产生的 refs
5. touchedResources 已经可以变成 file/url/resource refs

这次继续补齐：

1. 给 ContextRelation 增加 touched / modified / read / wrote 关系
2. appendRunResultRefs 里根据 touchedResources.operation 生成更准确的关系
3. TaskEnvelope Hydrator 支持 url/resource/file，并在 Files/Resources 中呈现
4. Previous Runs 显示 summary，防止没有 artifact 时内容丢失
5. 如果子 Agent 没有产出 artifact，但 final summary 有内容，自动生成 fallback artifact
6. 禁用或限制 TaskEnvelopeBuilder 里根据 agent 自动找最近 run，防止上下文串台
一、修改 ContextRelation 类型

文件：

agent-platform-node/src/modules/orchestration/orchestration.types.ts

当前 ContextRelation.relation 只有：

| "belongs_to"
| "created_by"
| "executed_by"
| "attempted_write"
| "attempted_write_artifact"
| "intended_for"
| "produced"
| "used_by"
| "derived_from";

请改成：

export type ContextRelation = {
  fromRefId: string;
  toRefId: string;
  relation:
    | "belongs_to"
    | "created_by"
    | "executed_by"
    | "attempted_write"
    | "attempted_write_artifact"
    | "intended_for"
    | "produced"
    | "used_by"
    | "derived_from"

    // 新增：资源操作关系
    | "touched"
    | "read"
    | "wrote"
    | "modified"
    | "deleted";
};

原因：

当前 touchedResources 统一转成 used_by，语义不准。
比如写文件应该是 wrote，读网页应该是 read，修改文件应该是 modified。

你的 ContextRefKind 现在已经支持：

| "file"
| "url"
| "resource"

这个是对的，保持即可。

二、修改 appendRunResultRefs：根据 operation 生成关系

文件：

agent-platform-node/src/modules/orchestration/orchestration.service.ts

当前 touchedResources 里大概是这样：

newRelations.push({
  fromRefId: refId,
  toRefId: runRefId,
  relation: "used_by" as const,
});

请改成：

const relation = mapTouchedOperationToRelation(resource.operation);

newRelations.push({
  fromRefId: runRefId,
  toRefId: refId,
  relation,
});

注意方向建议改成：

run -> resource

也就是：

run touched file
run wrote file
run read url

比现在的：

resource used_by run

更自然，更利于后续查“这个 run 动过什么”。

新增 helper：

function mapTouchedOperationToRelation(
  operation?: string
): import("./orchestration.types.js").ContextRelation["relation"] {
  const op = (operation || "").toLowerCase();

  if (["read", "fetch", "open", "visit", "navigate", "crawl", "scrape"].some((k) => op.includes(k))) {
    return "read";
  }

  if (["write", "create", "save", "append"].some((k) => op.includes(k))) {
    return "wrote";
  }

  if (["edit", "update", "modify", "patch"].some((k) => op.includes(k))) {
    return "modified";
  }

  if (["delete", "remove"].some((k) => op.includes(k))) {
    return "deleted";
  }

  return "touched";
}

同时 ref 的 summary 建议更明确：

summary: `${resource.operation || "touched"}; verified=${resource.verified ? "true" : "false"}`

status 也建议统一：

status: resource.verified ? "verified" : "unverified"

不要一会儿叫 ok，一会儿叫 success。

五、TaskEnvelopeRenderer 渲染 Resources

文件：

agent-platform-node/src/modules/orchestration/task-envelope-renderer.ts

在 Files 后面增加：

if (envelope.selectedContext.resources?.length > 0) {
  lines.push(`## Resources`);
  for (const res of envelope.selectedContext.resources) {
    lines.push(`- ${res.refId}`);
    lines.push(`  kind: ${res.kind}`);
    lines.push(`  uri: ${res.uri}`);
    if (res.lastKnownStatus) lines.push(`  status: ${res.lastKnownStatus}`);
    if (res.lastKnownOperation) lines.push(`  lastOperation: ${res.lastKnownOperation}`);
    if (res.summary) lines.push(`  summary: ${res.summary.slice(0, 300)}`);
  }
  lines.push("");
}

这样浏览器 Agent / 爬虫 Agent 访问过的 URL，也能被下游看见。

function inferAgentUidFromRunRef(ref: ContextRef): string {

七、没有 artifact 时自动生成 fallback artifact

这是最关键的补强。

文件：

agent-platform-node/src/runtime/agent-runtime.ts

当前 run 完成后会：

const agentResult = await buildAgentResult(...)

但如果这次 run 没有 agent_artifacts，producedArtifacts 为空，下游可能只能看到 run summary。

建议在 run 完成前或 buildAgentResult 前后加一个 fallback：

新增函数

可以放在 agent-runtime.ts 里，或者新增文件：

agent-platform-node/src/modules/artifacts/fallback-artifact.ts

示例：

async function persistFallbackArtifactFromFinalSummary(input: {
  workContextUid?: string;
  runId: number;
  summary: string;
  agentName: string;
}) {
  if (!input.workContextUid) return;
  if (!input.summary || input.summary.trim().length === 0) return;

  const existingArtifacts = await db
    .select({ id: agentArtifacts.id })
    .from(agentArtifacts)
    .where(eq(agentArtifacts.runId, input.runId))
    .limit(1);

  if (existingArtifacts.length > 0) return;

  // 根据 workContextUid 查 workContext 自增 id
  const [wc] = await db
    .select({ id: workContexts.id })
    .from(workContexts)
    .where(eq(workContexts.workContextUid, input.workContextUid))
    .limit(1);

  if (!wc) return;

  const now = nowIso();

  await db.insert(agentArtifacts).values({
    artifactUid: makeUid("artifact"),
    workContextId: wc.id,
    runId: input.runId,
    sourceRunId: input.runId,
    artifactType: "text",
    artifactRole: "intermediate",
    title: `${input.agentName} final output`,
    mimeType: "text/plain",
    contentText: input.summary,
    contentJson: "{}",
    status: "ready",
    sourceArtifactIdsJson: "[]",
    metadataJson: jsonStringify({
      generatedBy: "fallback_final_summary",
    }),
    createdAt: now,
    updatedAt: now,
  });
}

调用位置：

在 AgentRuntime.run() 中 executeRunLoop 返回 result 后，buildAgentResult 之前调用：

await persistFallbackArtifactFromFinalSummary({
  workContextUid: input.workContextId,
  runId: run.id,
  summary: result.summary,
  agentName: input.agentRecord.name,
});

然后再：

const agentResult = await buildAgentResult(...)

这样 buildAgentResult 查询 artifacts 时，就能把 fallback artifact 也纳入 producedArtifacts。

注意：只在没有 artifact 时生成，避免重复。

八、限制或移除 agent 自动扩散 refs

文件：

agent-platform-node/src/modules/orchestration/task-envelope-builder.ts

你现在有：

agent:xxx → findRelatedRefsForAgent(...)

建议先禁用，或者至少加 workContext 限制。

推荐最小安全改法

在 resolveRefs() 中，不要再对 agent 自动扩散：

if (directRef) {
  if (!addedRefIds.has(directRef.refId)) {
    resolvedRefs.push(directRef);
    addedRefIds.add(directRef.refId);
  }

  // 删除或注释这段：
  // if (directRef.kind === "agent") {
  //   const relatedRefs = this.findRelatedRefsForAgent(...)
  // }
}

原因：

agent 是执行者，不是数据源。
当前 plan 内数据传递已经由 dependsOn -> producedRefsByStepUid 解决。
继续通过 agent 找最近 run，可能拿到其他任务的历史 run，造成上下文串台。

如果你一定要保留，必须限制：

findRelatedRefsForAgent(agentUid, contextIndex, workContextUid)

只返回：

r.workContextUid === currentWorkContextUid

并且最好只在 inputRefIds 明确引用历史 agent 时才启用，不要默认启用。

九、更新验收标准
场景 1：文案 Agent 没有主动生成 artifact

用户：

生成2条小红书标题，然后让文件系统 Agent 写入文件，再让文案 Agent 根据标题扩写正文

即使第一个文案 Agent 只是 final summary 里输出标题，没有 artifact directive，也应该自动生成 fallback artifact。

下一个 Agent 的 TaskEnvelope 应该包含：

## Artifacts
- artifact:xxx: 文案Agent final output
  content:
  标题1...
  标题2...
场景 2：文件 Agent 写入文件

文件 Agent 的 touchedResources：

{
  "type": "file",
  "uri": "1.txt",
  "operation": "write",
  "verified": true
}

应生成：

ContextRef:
refId = file:1.txt
kind = file
status = verified

应生成关系：

run:xxx --wrote--> file:1.txt

不是：

file:1.txt --used_by--> run:xxx
场景 3：浏览器 / 爬虫 Agent 访问 URL

touchedResources：

{
  "type": "url",
  "uri": "https://example.com/page/1",
  "operation": "crawl",
  "verified": true
}

应生成：

ContextRef:
refId = url:https://example.com/page/1
kind = url

应生成关系：

run:xxx --read--> url:https://example.com/page/1

下游 TaskEnvelope 应显示：

## Resources
- url:https://example.com/page/1
  kind: url
  uri: https://example.com/page/1
  status: success
  lastOperation: crawl
场景 4：不能串台

如果 Step2 依赖 Step1，只能自动拿 Step1 的 produced refs。

不允许因为 Step2 的 inputRefIds 里出现 agent:writer_agent，就自动拉取 writer_agent 最近一次无关任务的 run。

十、最终要形成的链路

修改完成后，链路应该是：

SubAgent 执行
  ↓
ToolResult.operation / sideEffects / outputRefs
  ↓
AgentResult.touchedResources
  ↓
appendRunResultRefs
  ↓
生成 file/url/resource refs
  ↓
生成 run --read/wrote/modified/touched--> resource 关系
  ↓
executionContext.producedRefsByStepUid 记录这些 refs
  ↓
下一 step 根据 dependsOn 自动合并 refs
  ↓
TaskEnvelope Hydrator 查询 artifact/run/resource
  ↓
Renderer 渲染 Previous Runs / Artifacts / Files / Resources
  ↓
下一个 SubAgent 能继续干活
不要做的事
1. 不要重新设计 MainDecision.plan。
2. 不要引入复杂 selector。
3. 不要重新把 Original User Message 传给子 Agent。
4. 不要默认渲染 Selected Context。
5. 不要默认渲染完整 run steps。
6. 不要默认用 agentUid 去捞历史 run，容易串台。

核心一句话：

touchedResources 要进入 Resource Graph；artifact 负责内容传递，touchedResources 负责资源路径和副作用追踪。
