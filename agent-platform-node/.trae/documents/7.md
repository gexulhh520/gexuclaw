private buildStepOutcomeReviewSystemPrompt(): string {
  return `你是主 Agent 的 Step Outcome Reviewer，不是执行器。

你会收到：
1. 原始用户任务
2. 当前执行计划中的当前 step
3. 当前 TaskEnvelope
4. 子 Agent 执行后的 AgentResult
5. 系统基础规则验收 ruleEvaluation
6. 已完成 step 的简要结果

你的任务：
判断当前 step 是否可以被接受，以及下一步应该怎么做。

你必须只输出严格 JSON，不要 Markdown，不要代码块，不要解释性文字。

输出 JSON 格式：
{
  "decision": "continue | retry_same_agent | replan_remaining | ask_user | fail",
  "confidence": "high | medium | low",
  "safeToUseProducedRefs": true,
  "issues": ["string"],
  "retryInstruction": "string optional",
  "replanInstruction": "string optional",
  "userQuestion": "string optional",
  "finalMessage": "string optional",
  "reasoning": "string"
}

决策含义：
- continue：当前 step 可以接受，可以继续执行后续 step。
- retry_same_agent：当前 step 目标没有完成，但问题局部可修复，可以让同一个子 Agent 只修复当前 Objective。
- replan_remaining：当前计划、输入 refs、上下文或结果不可靠，需要重新规划当前 step 或剩余步骤。
- ask_user：缺少必须由用户确认的信息。
- fail：不可恢复失败。

核心判断规则：
1. 不要只相信子 Agent 的 final answer，必须优先相信工具事实、AgentResult、touchedResources、producedArtifacts、openIssues。
2. 如果 ruleEvaluation 有问题，必须结合语义判断是否真的影响后续。
3. 如果 expectedResultKind=artifact，但没有 producedArtifacts：
   - 如果只是忘记声明 artifact，且业务内容可靠，可以 retry_same_agent。
   - 如果业务内容本身不可靠或来源污染，replan_remaining。
4. 如果 expectedResultKind=file_change：
   - 工具事实证明目标资源被正确修改且 verified=true，可以 continue。
   - 如果只是日志结构缺失但工具事实足够明确，可以 continue，但 safeToUseProducedRefs 必须谨慎判断。
5. 如果子 Agent 使用了 TaskEnvelope 明确输入之外的资源替代指定输入，通常必须 replan_remaining，safeToUseProducedRefs=false。
6. 如果指定输入 refs 缺失、失效、读取失败，且子 Agent 又寻找其他文件替代，必须 replan_remaining，safeToUseProducedRefs=false。
7. 如果结果可能被错误上下文污染，必须 safeToUseProducedRefs=false。
8. retry_same_agent 只能用于局部格式修复或轻微遗漏，不能用于让 Agent 扩展任务范围。
9. retryInstruction 必须明确约束：只修复当前 Objective，不要执行其他 step，不要使用未指定资源替代。
10. replan_remaining 时，必须给出 replanInstruction，说明从哪里重新规划、哪些 refs 不可信。
11. 如果可以继续，safeToUseProducedRefs 必须为 true。
12. 如果 safeToUseProducedRefs=false，不应该 decision=continue。
13. reasoning 简短说明，不超过 300 字。`;
}

2. 暂时不要让 decideAutoRepair 决定主流程
3. 后续可以删除 decideAutoRepair

原因：

恢复动作由 MainAgent.reviewStepOutcome 决定。
编排层只负责执行主 Agent 的 decision，并加 retry 上限防死循环。
十二、最终执行流程

改完后流程应该是：

for each step:
  build TaskEnvelope
  runtime.run(subagent)
  AgentResultBuilder 生成 AgentResult
  evaluateStepResult 做基础检查
  MainAgent.reviewStepOutcome 做语义验收

  if continue && safeToUseProducedRefs:
      append refs
      next step

  if retry_same_agent:
      build retry envelope
      retry current step，受 MAX_STEP_RETRIES 限制

  if replan_remaining:
      stop current plan，返回主 Agent 重规划提示或失败信息

  if ask_user:
      返回用户问题

  if fail:
      返回失败信息

after all steps:
  MainAgent.summarizeFinalResult
  update main run final answer
十三、验收标准
1. 每个 step 后都会有主 Agent review 日志

应该看到类似：

[executePlanAsync] MainAgent step review: continue, safe=true, confidence=high

或者：

[executePlanAsync] MainAgent step review: replan_remaining, safe=false, confidence=high
2. 污染场景不会继续传递

场景：

历史 refs 里有 1.txt / 2.txt
用户手动删除本地 1.txt / 2.txt
文件 Agent 读取失败后尝试找其他文件替代

期望：

{
  "decision": "replan_remaining",
  "safeToUseProducedRefs": false,
  "issues": [
    "指定输入文件不可用",
    "子 Agent 使用了非 TaskEnvelope 指定资源替代，结果不可信"
  ]
}

后续文案 Agent 不应该继续执行。

3. artifact 格式问题可以 retry

场景：

文案 Agent 生成了标题文本，但没有 declaredArtifacts

期望：

{
  "decision": "retry_same_agent",
  "safeToUseProducedRefs": false,
  "retryInstruction": "只补充 declaredArtifacts，不要重新规划整个任务..."
}
4. 正常文件写入可以 continue

场景：

file_change step 写入 1.txt / 2.txt
AgentResult.touchedResources 有 file write verified=true

期望：

{
  "decision": "continue",
  "safeToUseProducedRefs": true
}
5. 最终用户回复不再拼接子 Agent summary

最终前端不应该看到：

1. agent_xxx：success
...
2. agent_yyy：success
...

应该看到主 Agent 统一总结，比如：

已完成：生成了 2 条小红书标题，并分别写入 1.txt 和 2.txt；随后根据标题扩写了正文并保存到对应文件中。
十四、最终原则
AgentResultBuilder：
  只整理执行事实。

evaluateStepResult：
  只做基础规则检查。

MainAgent.reviewStepOutcome：
  决定当前 step 是否可信，以及下一步 continue/retry/replan/ask_user/fail。

MainAgent.summarizeFinalResult：
  统一生成最终用户回复。

Orchestration：
  只负责执行主 Agent 的决策，并通过 retry 上限防死循环。

一句话：

子 Agent 负责做事，系统负责记录事实，主 Agent 负责验收和裁决，最终回复也由主 Agent 统一生成。
