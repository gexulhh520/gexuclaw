# Agent Platform 前端开发规范

本文档专门约束 Agent Platform 的前端实现方式。目标不是把接口堆到页面上，而是做出适合日常调试、查看运行记录、观察模型调用和后续多 Agent 工作流的专业工作台。

## 1. 总体目标

- 风格要专业、克制、可扫描，不做 demo 感页面。
- 页面要优先服务真实工作流：看 Agent、跑 Agent、看 Run、看 Steps、看 Model Invocation。
- 优先复用当前项目的 Vue 3 + TypeScript + Element Plus 技术栈。
- 保持与现有前端目录和路由风格一致，不另造一套前端架构。

## 2. 页面设计原则

- 第一屏就展示真实工作台，不做介绍页。
- 页面结构应优先采用三段式：
  - 顶部：标题、状态、刷新、主要动作。
  - 中部：主工作区，例如 Agent 列表、Run 列表、详情面板。
  - 侧边或底部：步骤、调用记录、JSON 摘要、错误信息。
- 信息密度适中，关键指标和状态优先，次要说明放辅助区。
- 操作按钮要明确区分主次，不让每个按钮都像主按钮。
- 所有列表、表格、时间线必须支持 loading / empty / error。

## 3. 视觉规范

- 采用专业工具后台风格：
  - 深浅分层清楚
  - 卡片少而精
  - 标题、说明、状态标签层级明确
  - 避免花哨渐变和装饰性背景
- 页面主背景和工作区背景需要有轻微层次，但不要做成营销页。
- 重点信息用数字、标签、分组标题和细边框增强扫描效率。
- 长 JSON、日志、步骤内容要用等宽字体展示。
- 详情面板优先使用 Split Panel / Drawer / Tabs，而不是把所有信息堆在一张长表里。

## 4. 组件规范

- 页面组件负责数据编排和布局。
- 业务组件负责单一职责，例如：
  - `AgentSummaryCard`
  - `RunTimeline`
  - `InvocationPanel`
  - `MetricStrip`
- API 调用统一放到 `frontend/src/api/agentPlatform.ts`。
- 不在多个页面里重复写相同的格式化逻辑。
- 任何展示 JSON 的组件都应支持滚动和固定高度，避免撑坏页面。

## 5. Agent Platform 第一版页面范围

第一版只做最小可用工作台：

- Agent 总览
- Agent 列表
- Agent 当前版本信息
- 运行入口
- 最近 Run 列表
- Run Steps
- Model Invocations

暂不做：

- 多 Agent 编排画布
- WorkContext 可视化
- Artifact 管理台
- Memory 管理
- 项目空间管理

## 6. API 使用规范

- Agent Platform API 单独封装，不混进现有聊天 API。
- 本地开发默认通过 Vite proxy 转发到 `http://127.0.0.1:3100`。
- 所有接口返回都走统一解包函数。
- 错误提示要给用户可理解的中文说明，不直接把原始异常甩到 UI。

## 7. 状态规范

每个页面和关键区域都要考虑：

- 初次加载状态
- 手动刷新状态
- 空列表状态
- 请求失败状态
- 正在运行状态
- 最近一次执行成功 / 失败状态

## 8. 实现约束

- 使用 Vue 3 Composition API。
- 使用 TypeScript。
- 样式优先写在单文件组件内，通用样式再抽离。
- 不为第一版引入新的重型 UI 库。
- 新增页面必须同步更新开发进度记录。
