## 问题归因

- 根因不是 `scheduled_task__plan_draft` 没有被调用。
- 根因是 V2 异步草案链路未完整透传 `provider / model`，导致后台 `PlannerAgentV2` 可能使用错误的模型提供方。
- 当前前端已经兼容 `planner_draft_queued / planner_draft_ready / planner_draft_failed`，因此更可能是后台任务本身执行失败或输出异常，而不是前端事件类型不识别。
- 此外，V2 worker 收尾逻辑相对旧版存在功能回退：执行步骤元数据减少、turn 与 source user message 关联缺失、每轮对话结束后不再触发 turn memory 构建与向量索引。
- 异步 planner job 链路当前可运行，但可观测性不足；即使任务已成功投递，如果 worker 日志窗口没有展示 Celery 的标准 received/succeeded 行，也无法从业务日志确认 plan 是否真正开始执行。
- 在 Windows `solo` worker 开发模式下，聊天主任务、planner 草案任务、turn memory 任务共用同一执行资源，会造成“首轮返回后后台任务立刻占住 worker，第二轮聊天像是被堵塞”的体验问题。
- 目前需要通过 A/B 实证进一步区分：真正的主要阻塞源是新增 `planner job`，还是恢复后的 `turn_memory` 任务，抑或两者叠加。
- 当前 `planner` 还有一个独立稳定性问题：`_build_trace_context()` 依赖的记忆召回链路并非草案生成的必需条件，但目前异常会直接中断整个 planner，需要降级为 warning + 空上下文回退。
- `PlannerAgentV2` 还存在消息格式兼容性问题：Agent 层混用了 LangChain Message 对象与项目内自定义 LLM client 的字典消息格式，导致 planner 真正调用模型时发生类型错误。

## 修复计划

1. 在 `backend/agents/executor_v2.py` 的异步投递节点中补齐 `provider / model` 透传。
2. 在 `backend/workers/planner_jobs.py` 任务签名与内部 `_generate()` 中接收并透传 `provider / model`。
3. 在 `backend/services/planner_service_v2.py` 继续沿用传入的 `provider / model` 初始化 `PlannerAgentV2`，确保与主聊天会话一致。
4. 补充一个针对异步投递链路的验证脚本，验证 tool call 命中后入队参数包含正确的 `provider / model`。
5. 执行诊断检查，确认改动未引入新的语法或类型错误。
6. 补齐 `backend/workers/tasks_v2.py` 的收尾逻辑：恢复 `source_user_message_id` 回写、补回 turn memory 任务触发、丰富 execution steps metadata，并按旧版顺序先落步骤再完成 turn。
7. 增加一个针对 V2 worker 收尾行为的验证脚本，确认步骤落库、turn 完成、source message 关联与 turn memory 触发全部正常。
8. 补充 `executor_v2 -> planner_jobs` 的关键日志，至少覆盖“已投递 / 已开始 / 已成功 / 已失败”四个节点，便于在 Terminal 直接判断 plan 是否真正进入后台执行。
9. 针对开发环境单 worker 堵塞问题，增加后台任务短延迟入队策略，避免第一轮结束后立即抢占下一轮聊天的执行时机。
10. 同步补充 worker 启动建议：开发环境至少拆分“聊天 worker”和“后台 worker”，减少 `solo` 模式下的串行阻塞。
11. 新增 `planner dispatch` A/B 开关，允许在不改前端调用方式的情况下临时禁用 `executor_v2` 的后台草案投递，用于真实环境复现“第二轮是否恢复”。
12. 修复 `_build_trace_context()` 的健壮性：记忆召回、步骤读取等可选链路异常时不再抛出到最外层，而是降级为空结果并继续生成草案。
13. 修复 `PlannerAgentV2` 的消息格式：调用 `llm_client_instance.chat()` 时统一使用项目内部约定的 `dict` 消息结构，避免 `HumanMessage` 类型错误。

## 验证要点

- 主会话使用 `kimi` 时，异步草案任务不能回退到 `openai`。
- `planner_draft_queued` 后，后台任务应能稳定产出 `planner_draft_ready` 或清晰的 `planner_draft_failed`。
- 已有同步链路与前端卡片展示行为不应被破坏。
- V2 每轮对话结束后应恢复触发 `build_turn_memory_task`，以生成 turn memory 与向量索引。
- V2 保存的 execution steps 应至少保留 `draft_id / draft_title / analysis_status / intent_text / request_id / job_id` 等排障与回放字段。
- 当主任务返回“后台开始生成草案”后，Terminal 应至少能看到 planner job 的投递日志和后台开始执行日志，避免误判。
- Windows `solo` 模式下，第一轮结束后不应因为 planner / memory 任务立刻开跑而明显阻塞第二轮聊天；至少要为紧随其后的聊天请求留出抢占窗口。
- 当关闭 `planner dispatch` A/B 开关后，如果第二轮恢复正常，则可基本确认新增 `planner job` 是主要阻塞源。
- `planner` 进入执行后，即便 `memory_retrieval_service` 或轨迹构建局部失败，也应至少返回基于最近消息和 execution steps 的降级草案，而不是直接报 `IndexError`。
- `planner` 调模型时应统一使用 `{"role": ..., "content": ...}` 消息格式，不能把 LangChain 的 `HumanMessage` 直接传给项目内 LLM client。
