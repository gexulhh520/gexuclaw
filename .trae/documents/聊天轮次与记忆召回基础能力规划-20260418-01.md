# 聊天轮次与记忆召回基础能力规划

## 1. 背景

当前项目已经具备：

- `chat_sessions`
- `chat_messages`
- `agent_execution_steps`
- LangGraph 主图与定时任务子图骨架

但当前仍缺少两个关键基础能力：

1. 缺少“轮次（turn）”这一等建模能力  
   目前 `chat_messages` 与 `agent_execution_steps` 没有明确的 `turn_id`，很多逻辑只能靠消息顺序推导。

2. 缺少“历史消息/流程的可召回记忆层”  
   当前定时任务子图获取上下文时，仍主要依赖会话内消息回查与执行步骤遍历，不适合支持“几轮之前”“很久之前”的流程转定时任务。

因此需要先补齐基础能力，再继续推进“主图方案1：主 agent 通过工具触发子图”的方案。

## 2. 本期目标

本期只做基础能力，不直接做主图方案1。

本期目标：

- 建立明确的聊天轮次模型
- 支持每轮结束后的静默异步记忆沉淀
- 建立“最近上下文 + 向量召回历史消息/流程”的上下文获取基础
- 为后续定时任务子图、主图工具触发方案提供稳定上下文层

## 3. 范围边界

### 3.1 本期包含

- 新增 `chat_turns` 表
- 为 `chat_messages`、`agent_execution_steps` 增加 `turn_id`
- 每轮结束后异步沉淀 turn memory
- turn memory 的结构化摘要存储
- turn memory / workflow snapshot 的向量化入库
- 提供基础召回接口给后续子图使用

### 3.2 本期不包含

- 不直接改成“主 agent tool call 触发定时任务子图”
- 不在本期做完整 rerank 排序系统
- 不在本期做 workflow snapshot 的复杂 UI 展示
- 不考虑老数据迁移兼容

## 4. 设计原则

### 4.1 轮次是一等概念

后续所有能力都应以轮次为基本聚合单位：

- 每轮的 user message
- 每轮的 assistant 最终消息
- 每轮的工具执行步骤
- 每轮的结果摘要
- 每轮的记忆 embedding

### 4.2 原始事实与召回记忆分层

- 原始事实保留在 `PostgreSQL`
- 可检索的摘要与向量独立存储
- 向量库只存“可召回摘要”，不存整轮全量原始消息

### 4.3 上下文使用双通道

后续子图获取上下文时必须采用：

- 少量最近上下文
- 向量召回的历史消息/流程摘要

不能只依赖最近窗口，也不能直接把全量历史塞进 LLM。

### 4.4 表结构新增字段单独执行

按当前项目规则：

- 涉及数据表新增字段、新增表等
- 通过单独的 test/migration 脚本执行
- 不把数据迁移逻辑写入启动主链路

## 6. 运行时链路规划

### 6.1 每轮开始

在用户发送消息并进入主图执行前：

1. 创建 `chat_turn`
2. 状态为 `running`
3. 生成本轮 `turn_index`
4. 创建 user message 时写入 `turn_id`

### 6.2 每轮执行中

主图运行时：

- 所有 `agent_execution_steps` 写入时带 `turn_id`
- 若需要中间 assistant/system 消息，也写 `turn_id`

### 6.3 每轮结束

assistant 最终结果生成后：

1. 创建 assistant message，并写入 `turn_id`
2. 更新 `chat_turn.assistant_message_id`
3. 更新 `chat_turn.status=completed`
4. 记录 `finished_at`
5. 异步投递静默记忆任务

### 6.4 每轮失败

若主图执行失败：

1. 更新 `chat_turn.status=failed`
2. 写失败时间
3. 可选生成失败类 turn memory，但默认不入 workflow snapshot

## 7. 静默记忆任务规划

### 7.1 任务触发时机

在一轮结束后异步触发，不阻塞主聊天响应。

任务建议名：

- `build_turn_memory_task`

### 7.2 输入

- `turn_id`

通过 `turn_id` 再去查：

- user message
- assistant message
- 本轮 execution steps
- session / knowledge_base_ids
- user settings

### 7.3 输出

生成一条 `turn_memories` 记录：

- 用户目标摘要
- 最终结果摘要
- 工具执行链摘要
- 成功失败工具
- 是否适合作为 task candidate
- 召回标签

### 7.4 向量化内容

embedding 文本建议只包含：

- `user_query`
- `task_goal`
- `result_summary`
- `tool_trace_summary`
- `tags`

不直接存：

- 全量 tool result
- 全量页面 markdown
- 全量 chain-of-thought
- 大段原始 JSON

## 8. 上下文召回策略规划

### 8.1 近端上下文

后续子图必须取少量最近上下文：

- 最近 `6~12` 条消息
- 最近 `1~3` 轮 turn
- 最近一轮成功执行摘要

### 8.2 历史召回

通过向量召回历史 turn memory：

- 默认 `top_k=5~8`
- 再进行去重与裁剪

### 8.3 后续工作流快照召回

当 `workflow_snapshots` 建好后，优先召回 snapshot，再补 message/turn memory。

### 8.4 合并输入

后续定时任务子图输入建议结构：

- `recent_context`
- `retrieved_turn_memories`
- `retrieved_workflow_snapshots`
- `route_context`
- `user_settings`

## 9. 分阶段实施步骤

### 阶段 1：轮次建模

目标：

- 建立 `chat_turns`
- `chat_messages.turn_id`
- `agent_execution_steps.turn_id`

步骤：

1. 新增模型 `chat_turn.py`
2. 更新 `chat_session.py`、`agent_execution_step.py`
3. 调整消息写入逻辑
4. 调整执行步骤写入逻辑
5. 新增独立 test 脚本执行建表/补字段

### 阶段 2：轮次闭环

目标：

- 每轮开始、执行、结束都有明确 turn 生命周期

步骤：

1. 用户消息创建前初始化 `chat_turn`
2. assistant 最终消息完成后补齐 `assistant_message_id`
3. 执行失败场景补 `failed`
4. 校验一轮只对应一个最终 assistant 结果

### 阶段 3：turn memory 静默任务

目标：

- 每轮结束后自动生成结构化记忆

步骤：

1. 新增 `turn_memories` 模型
2. 新增 `build_turn_memory_task`
3. 根据 `turn_id` 聚合本轮内容
4. 生成结构化摘要并入库

### 阶段 4：向量化

目标：

- turn memory 支持向量召回

步骤：

1. 设计向量 schema
2. 新增 embedding 写入服务
3. turn memory 完成后异步写向量
4. 加幂等键避免重复写入

### 阶段 5：基础召回接口

目标：

- 提供给后续子图使用的统一召回服务

步骤：

1. 新增 `memory_retrieval_service`
2. 提供 recent turn 查询
3. 提供 vector retrieval
4. 提供 recent + retrieved 合并接口

### 阶段 6：为方案1铺路

目标：

- 主 agent 工具触发子图时可直接用这套上下文层

步骤：

1. 主图工具调用只传核心参数
2. 子图通过 `turn_id/session_id` 获取 recent context
3. 子图通过 memory retrieval 获取历史召回
4. 用于定时任务草案规划

## 10. 建表与字段执行规则

按当前约束执行：

- 所有新增表/字段通过独立 test 脚本执行
- 不在 `main.py` 写迁移
- 不考虑老数据回填

建议脚本拆分：

- `backend/test_scripts/create_chat_turn_schema.py`
- `backend/test_scripts/create_turn_memory_schema.py`

## 11. 验证标准

### 11.1 轮次模型验证

- 一条 user message 必须有 `turn_id`
- 一条 assistant 最终消息必须有 `turn_id`
- 本轮 execution steps 必须有 `turn_id`
- `chat_turns` 能唯一聚合这一轮

### 11.2 静默记忆验证

- 每轮结束后能异步生成一条 turn memory
- turn memory 不阻塞主聊天响应
- turn memory 摘要不包含大段无用原始工具返回

### 11.3 向量化验证

- 同一 `turn_id` 不重复写向量
- 能用用户当前意图召回几轮前的相关 turn memory

### 11.4 后续子图验证前提

- 近端上下文与历史召回能稳定同时提供
- 为主图方案1提供可复用基础

## 12. 下一步顺序

建议严格按下面顺序推进：

1. 先做 `chat_turns + turn_id`
2. 再做 turn 生命周期闭环
3. 再做 `turn_memories`
4. 再做向量化
5. 最后再规划和落地主图方案1（主 agent tool 触发子图）

不要跳过基础层直接改主图方案1，否则子图上下文效果仍然不稳定。
