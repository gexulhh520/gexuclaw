## 一、背景

当前浏览器工具运行在 Agent Worker 进程内部，链路为：

1. Agent Worker
2. `BrowserTool`
3. `TaskBrowserManager`
4. Playwright
5. Chrome / Chromium 子进程

在 Windows + Celery Worker 场景下，当前事件循环策略需要兼容异步 checkpoint，而 Playwright 启动浏览器又需要支持 subprocess 的事件循环。两者存在冲突。

因此本次不再尝试通过调整同一 Worker 进程内的事件循环策略来兼容，而是将浏览器能力独立出去。

---

## 二、目标

本次改造目标：

1. 浏览器能力运行在独立进程/独立服务中
2. Agent Worker 不再直接持有 Playwright Browser 实例
3. `BrowserTool` 改为通过 HTTP 调用浏览器服务
4. 保持现有工具名不变，尽量不影响 LLM 工具调用协议
5. 允许后续将浏览器服务单独部署和扩容

---

## 三、总体架构

### 3.1 改造后结构

1. Agent Worker
2. `BrowserTool` HTTP Client
3. Browser Service（独立 FastAPI 进程）
4. Browser Session Manager
5. Playwright
6. Chrome / Chromium

### 3.2 关键原则

- Agent Worker 只负责业务编排，不直接操作浏览器
- 浏览器服务自己管理 session 生命周期
- 浏览器服务与 Agent Worker 通过 HTTP JSON 协议通信
- 现有浏览器操作接口尽量保持语义不变

---

## 四、第一阶段范围

### 4.1 第一阶段要做

- 新增独立浏览器服务应用
- 新增浏览器服务客户端
- `BrowserTool` 支持通过服务端执行浏览器操作
- 配置项支持切换是否启用浏览器服务
- 支持以下核心操作：
  - `launch_browser`
  - `navigate`
  - `click`
  - `fill`
  - `type`
  - `press`
  - `get_text`
  - `get_page_info`
  - `get_page_markdown`
  - `close_session`

### 4.2 第一阶段暂不做

- 独立认证鉴权
- 浏览器服务高可用
- 多节点 session 共享
- 分布式 session 存储

## 五、服务接口设计

### 5.1 通用接口

- `POST /browser/execute`

请求体：

```json
{
  "operation": "launch_browser",
  "args": {
    "browser_session_id": "browser_001",
    "url": "https://example.com"
  }
}
```

响应体：

```json
{
  "success": true,
  "data": {},
  "error": null,
  "meta": {}
}
```

### 5.2 健康检查

- `GET /health`

用于：

- Agent Worker 判断浏览器服务是否可用
- 部署后做基础探活

## 七、切换策略

### 7.1 配置项

建议新增：

- `BROWSER_SERVICE_ENABLED`
- `BROWSER_SERVICE_URL`
- `BROWSER_SERVICE_HOST`
- `BROWSER_SERVICE_PORT`

### 7.2 运行逻辑

- 若 `BROWSER_SERVICE_ENABLED=true`，`BrowserTool` 通过 HTTP 调用浏览器服务
- 若未启用，则保留当前本地模式，方便开发环境过渡

## 九、第一阶段验收标准

1. 浏览器服务可以独立启动
2. `BrowserTool.launch_browser` 在启用服务模式时可成功发起 HTTP 调用
3. `launch_browser -> navigate -> get_page_markdown -> close_session` 可打通
4. Agent Worker 不再直接在同一进程内创建 Playwright 子进程
