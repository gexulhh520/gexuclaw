# 定时任务模块补充实施计划

## Summary

本计划在现有 `定时任务模块规划方案.md` 基础上补齐高影响产品决策，并把“聊天中识别定时意图 -> 用户手动确认发起分析 -> 定时任务评估子 Agent 生成结构化草案 -> 聊天卡片展示 -> 弹窗二次确认 -> 预检查预览 -> 持久化任务 -> 调度执行 -> 站内/邮件/微信通知”拆成可直接实施的后端、前端、调度、通知与用户设置改造。

2026-04-18 需求变更同步：

- 不再使用规则/正则做定时任务意图检测
- 不再采用“用户点击开始分析后，普通 service 调一次 LLM”的低配方案
- 改为：主 agent 通过专用 tool call 判定并进入“定时任务规划子图”
- 定时任务规划必须实现为真正的 LangGraph 子图 Agent
- 子图在聊天执行阶段自动产出草案，用户只对最终草案做确认/预览/创建

本次补充后，第一版的明确目标是：

- 主聊天 Agent 负责轻量识别“定时执行意图”，但不直接落库
- 用户先在聊天里手动确认“开始分析”
- 启动专用定时任务评估子 Agent，对历史执行轨迹做最佳路径提炼、风险评估、依赖检查、自愈策略生成
- 评估结果先以聊天卡片展示，再进入弹窗做最终确认
- 用户只允许修改元信息：任务名称、描述、频率、时区、通知方式
- “预览执行”第一版以预检查优先，必要时仅做安全只读探测，不做完整高风险试跑
- 正式创建后由数据库驱动调度器执行，并生成执行历史与通知记录
- 首批通知通道必须真实可用：站内、邮件、个人微信

  - 后续如再涉及字段升级，改为单独迁移/test 脚本执行，不入侵启动主链路

### 本轮新增决策

- 定时任务触发方式：
  - 主聊天 Agent 不再用正则做检测
  - 改为由主 agent 的 LLM 决定是否调用 `scheduled_task__plan_draft`
  - 一旦进入子图，由子图自动完成草案分析与生成
- 评估方式：
  - 必须采用 LangGraph 子图 Agent，而不是单独 service 直调一次 LLM
  - 子图负责最佳路径提炼、风险分析、可行性评估、自愈策略、确认文案生成
- 最终确认入口：
  - 聊天卡片 + 弹窗双层确认
- 弹窗可编辑范围：
  - 仅元信息可编辑，执行步骤只读
- 预览执行语义：
  - 第一版以预检查优先
  - 仅允许安全只读探测，不做完整高风险试跑
- 通知通道：
  - 第一版必须交付站内 + 邮件 + 个人微信
- 通知配置来源：
  - 用户设置页维护默认邮箱、微信配置、默认时区
- 个人微信通道：
  - 通过 ClawBot 底层协议接口适配
  - 该通道属于外部依赖，需单独做配置校验、降级与失败隔离

### 需要补充的产品点

- 草案生命周期需要从现有 `draft / active / paused` 扩展为更完整的状态语义：
  - `analysis_pending`
  - `analysis_running`
  - `draft`
  - `preview_ready`
  - `active`
  - `paused`
  - `failed`
  - `archived`
- 需要显式区分三类执行结果：
  - 评估结果
  - 预览结果
  - 正式运行结果
- 需要补一套用户通知配置模型，否则“三通道都可用”无法真正落地
- 需要补“任务幂等键/去重策略”，避免 dispatcher 多次扫描造成重复入队
- 需要补“草案过期策略”，避免聊天里生成的历史草案长期堆积

## Proposed Changes

### 1. 聊天意图识别与评估子 Agent 编排

#### 涉及文件

- `backend/api/v1.py`
- `backend/workers/tasks.py`
- `backend/agents/executor.py`
- `backend/agents/scheduled_task_planner_subgraph.py`
- `backend/services/scheduled_task_planner_service.py`
- `backend/schemas/scheduled_task_planner.py`
- `backend/api/scheduled_tasks.py`

#### 变更内容

- 在主 LangGraph 中新增专用工具：
  - `scheduled_task__plan_draft`
  - 参数仅包含 `intent_text / schedule_text / timezone / goal`
- 当主 agent 的 tool call 命中该工具时，主图进入 `scheduled_task_planner_subgraph`
- 新增真正的 LangGraph 子图：
  - Node 1：提取任务意图与调度语义
  - Node 2：加载历史会话、工具成功/失败轨迹、知识库范围、用户通知配置
  - Node 3：规划最佳执行路径
  - Node 4：生成风险评估、预检查与自愈策略
  - Node 5：生成结构化草案与用户摘要
- 子图执行完成后：
  - 自动创建 `draft`
  - 向聊天前端发送“草案已生成”的事件
  - 不再要求用户先点“开始分析”
- `scheduled_task_planner_service` 退化为子图内部的领域服务，负责持久化与结构化封装，不再承担“子 agent 本体”角色

#### 为什么这样改

- 用户已明确要求“主 agent 的 llm 知道可以调用工具去生成定时任务”
- 这让“是否进入任务规划流程”由主 agent 的 function calling 决策，而不是额外 router 补丁

## Assumptions & Decisions

- 第一版保留现有 `scheduled_task` 骨架，不推翻重做；以补齐状态机、真实执行与通知链路为主
- 主聊天只负责检测与发起，不直接承担定时任务完整规划
- 主 agent 的 tool call 决定是否进入定时任务规划子图
- 定时任务评估必须由 LangGraph 子图 Agent 输出结构化结果
- 定时任务子图内部使用与主 Agent 相同的 `provider/model`，不单独硬编码 provider
- 主系统提示词需明确 `scheduled_task__plan_draft` 的触发条件、禁止场景与参数约束
- 聊天确认流程采用“建议卡片 -> 分析完成卡片 -> 弹窗最终确认”
- 执行步骤第一版只读，避免用户直接篡改工具链路
- 预览执行第一版默认不做完整副作用试跑，只做预检查和必要只读探测
- 调度基准仍以用户时区为准；由于当前 `User` 无时区字段，本轮必须补用户时区设置
- 邮件和个人微信都属于首批真实交付，不仅是预留字段
- 个人微信采用 ClawBot 协议适配，但必须通过统一通知服务隔离，避免外部依赖污染核心调度逻辑
- 若个人微信配置缺失或外部接口不可用，任务正式执行结果可成功，但该通道通知应单独失败并落明细
- 当前环境不在 `main.py` 启动阶段执行字段迁移；启动阶段只负责新环境建表

- 注册 `scheduled_tasks` 路由与用户设置路由

### T2 子 Agent 评估链路

- 主 agent 调用 `scheduled_task__plan_draft`
- 子图自动生成结构化草案和聊天确认卡片
- 用户不再手动触发分析，只对草案做确认

### T3 调度与通知链路

- dispatcher/executor 真正接入 Celery
- 真实执行工具步骤
- 打通站内、邮件、个人微信通知

### T4 前端管理与验收

- 聊天卡片、确认弹窗、任务列表页、设置页完成
- 完成端到端联调与失败场景验证

## Verification Steps

1. 聊天触发验证
   - 普通聊天不应误调用 `scheduled_task__plan_draft`
   - 主 agent 调用该 tool 时，主图正确进入子图
   - 子图完成后聊天中出现“草案已生成”卡片

2. 子 Agent 评估验证
   - 主图进入子图后生成 `analysis_running -> draft`
   - 草案包含：名称、描述、频率、cron、时区、执行步骤、成功条件、风险、可行性、通知方式
   - 草案优先选择历史成功路径并剔除失败步骤

3. 预览执行验证
   - 预检查输出 blockers/warnings/checks
   - 未配置邮箱或微信时，若用户勾选相应通知方式，应给出明确阻塞原因
   - 浏览器服务关闭、工具不在白名单、参数缺失时，能给出明确不可行原因

4. 创建任务验证
   - 用户仅能修改元信息
   - 确认创建后任务状态变为 `active`
   - 任务保存执行计划快照和原始轨迹

5. 调度执行验证
   - dispatcher 只派发一次同一到期任务
   - executor 能复用现有工具运行时执行步骤
   - 成功与失败都正确更新 `next_run_at`、`failure_count`、`auto_pause_reason`

6. 通知验证
   - 站内通知可在任务模块查看
   - 邮件通知可投递到用户设置页配置的邮箱
   - 个人微信通知可通过 ClawBot 通道投递
   - 单一通知通道失败不会误标整个任务执行失败

7. 前端页面验证
   - `/tasks` 可查看列表、详情、运行记录、通知明细
   - `/settings` 可维护时区、邮箱、微信配置
   - `Chat.vue` 中“我的任务”入口可正确跳转
