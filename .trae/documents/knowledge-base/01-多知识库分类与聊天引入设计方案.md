## 一、背景与目标

当前系统已经具备基础的文档上传、解析、向量化和知识检索能力：

- 前端聊天页面已有悬浮菜单入口，适合作为“我的知识库”入口承载点
- 后端已支持文档上传与异步处理
- 向量检索已基于 LanceDB + Ollama Embedding 跑通

但当前知识库的组织方式仍然是“每个用户一个总知识库”，还不支持：

- 用户创建多个知识库
- 按分类管理知识库
- 在任意聊天窗口引入指定知识库
- 仅针对选中的知识库做检索

本次开发目标是将现有能力升级为“多知识库体系”，支持用户在前端创建和管理多个知识库，并在聊天会话中灵活引入一个或多个知识库。

---

## 二、核心设计结论

### 2.1 设计原则

本次不采用“单知识库 + 文件标签”模式，而采用“多知识库实例”模式。

原因：

- 后续聊天场景需要明确“本次会话引用哪些知识库”
- 知识库删除、重建、统计、检索边界更清晰
- 未来更容易扩展共享、权限、模板库、团队库等能力

### 2.2 核心实体

建议引入以下 4 层结构：

1. **分类**
   - 用于组织知识库，例如：法律法规、案例库、内部制度、客户资料
2. **知识库**
   - 用户创建的具体库，例如：合同审查库、劳动法库、公司制度库
3. **文档**
   - 上传到某个知识库中的文件
4. **向量分块**
   - 文档解析后的 chunk，检索时直接命中这一层

### 2.3 目标使用路径

用户使用路径应为：

1. 打开聊天页悬浮球
2. 进入“我的知识库”
3. 创建分类或直接创建知识库
4. 向指定知识库上传文档
5. 在聊天窗口勾选要引入的知识库
6. 发送问题，AI 仅在选中的知识库内检索并回答

## 三、范围定义

### 3.1 第一阶段必须支持

- 用户创建多个知识库
- 每个知识库有名称、分类、描述
- 文档上传时必须归属某个知识库
- 在聊天窗口中引入一个或多个知识库
- 检索仅命中选中的知识库
- 回答中返回来源信息（知识库名、文件名、片段）

### 3.2 第一阶段暂不支持

- 知识库共享给其他用户
- 团队级知识库
- 自动分类归档
- 文档版本历史
- 检索排序可视化调参
- 知识库权限细粒度控制

## 五、数据模型设计

### 5.1 数据库表建议

建议新增关系型表来管理知识库元数据，不要只依赖 LanceDB。

#### 表 1：knowledge_bases

建议字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int / uuid | 主键 |
| user_id | int | 所属用户 |
| name | varchar | 知识库名称 |
| category | varchar | 分类名称 |
| description | text | 描述 |
| is_default | bool | 是否默认知识库 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

约束建议：

- 同一用户下知识库名称唯一
- `user_id + category` 建索引

#### 表 2：knowledge_documents

建议字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int / uuid | 主键 |
| knowledge_base_id | int / uuid | 所属知识库 |
| user_id | int | 所属用户 |
| original_filename | varchar | 原始文件名 |
| stored_filename | varchar | 存储文件名 |
| file_path | varchar | 文件路径 |
| file_type | varchar | 文件类型 |
| status | varchar | pending/processing/success/failed |
| chunks_count | int | 分块数量 |
| error_message | text | 错误信息 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 5.2 向量分块字段扩展

`DocumentChunk` 建议新增字段：

| 字段 | 说明 |
|------|------|
| knowledge_base_id | 知识库 ID |
| knowledge_base_name | 知识库名称 |
| category | 分类 |
| document_id | 文档 ID |
| created_at | chunk 创建时间 |

### 5.3 LanceDB 存储策略

推荐保留“每个用户一个 LanceDB table”的大方向，但在 chunk 元数据中加入 `knowledge_base_id`。

优点：

- 兼容现有按用户隔离的设计
- 无需把 LanceDB 拆成每库一张表，避免表数量快速膨胀
- 支持同用户内按知识库过滤检索

检索时通过：

- `user_id`
- `knowledge_base_id in (...)`

来过滤。

## 六、后端接口设计

### 6.1 知识库管理接口

建议新增文件：

- `backend/api/knowledge_bases.py`

建议接口如下：

#### 1. 创建知识库

`POST /api/knowledge-bases`

请求体：

```json
{
  "name": "合同审查库",
  "category": "法律业务",
  "description": "用于合同审查与条款比对"
}
```

#### 2. 获取知识库列表

`GET /api/knowledge-bases`

支持返回：

- 按分类分组
- 每个知识库的文档数
- 每个知识库的分块数

#### 3. 获取知识库详情

`GET /api/knowledge-bases/{knowledge_base_id}`

#### 4. 更新知识库

`PUT /api/knowledge-bases/{knowledge_base_id}`

#### 5. 删除知识库

`DELETE /api/knowledge-bases/{knowledge_base_id}`

删除时应同步处理：

- `knowledge_documents` 元数据
- LanceDB 中该知识库对应的所有 chunk
- 本地已上传的原始文件

### 6.2 知识库文档接口

#### 1. 上传文档到指定知识库

`POST /api/knowledge-bases/{knowledge_base_id}/documents`

说明：

- 复用现有文档上传逻辑
- 上传时必须绑定 `knowledge_base_id`
- 异步处理完成后更新 `knowledge_documents.status`

#### 2. 获取知识库文档列表

`GET /api/knowledge-bases/{knowledge_base_id}/documents`

#### 3. 删除单个文档

`DELETE /api/knowledge-bases/{knowledge_base_id}/documents/{document_id}`

需要同步删除：

- 关系表文档记录
- LanceDB 中该文档 chunk
- 上传的原始文件

### 6.3 检索接口或工具扩展

现有知识检索需要新增过滤参数：

- `knowledge_base_ids`

建议支持：

```json
{
  "query": "劳动合同解除条件",
  "knowledge_base_ids": [1, 2],
  "top_k": 8
}
```

检索返回建议包含：

- 命中文本
- 知识库名称
- 分类
- 文件名
- chunk 序号

## 七、聊天集成设计

### 7.1 会话级引入机制

每个聊天会话需要保存当前引入的知识库列表。

建议在会话维度新增字段：

- `knowledge_base_ids`

用途：

- 用户切换会话后仍保留该会话的知识库上下文
- 新消息发送时自动携带已引入知识库

### 7.2 消息发送逻辑

前端发送消息时建议带上：

```json
{
  "content": "请根据我的知识库分析这份合同风险",
  "provider": "deepseek",
  "model": "deepseek-chat",
  "knowledge_base_ids": [1, 3]
}
```

### 7.3 Agent 检索逻辑

Agent 在决定是否调用知识检索工具时：

- 若当前会话未引入任何知识库，则不调用知识库检索
- 若已引入知识库，则检索范围仅限会话已引入的库

## 八、前端交互设计

### 8.1 入口设计

在 `Chat.vue` 悬浮菜单中新增：

- 我的知识库

点击后打开右侧抽屉，不建议先做单独路由页面。

原因：

- 轻量
- 与聊天场景联动更自然
- 切换成本低

### 8.2 抽屉结构建议

#### 区块 1：知识库列表

- 按分类分组
- 显示知识库名称、文档数、更新时间
- 支持新建、编辑、删除

#### 区块 2：文档管理

- 展示当前知识库下的文档列表
- 显示处理状态、分块数、失败原因
- 支持上传、删除

#### 区块 3：检索测试

- 输入一个问题
- 返回命中的片段列表
- 显示来源知识库和文件名

### 8.3 聊天区联动

在聊天输入区域上方增加：

- 当前已引入知识库条

交互建议：

- 可快速勾选/取消知识库
- 可显示“已引入 2 个知识库”
- 可直接跳转到知识库管理抽屉

## 十、风险与注意事项

### 10.1 数据迁移

由于当前系统已经有旧的“用户单库”向量数据，开发时需要确定迁移策略：

- 方案 A：旧数据全部归入“默认知识库”
- 方案 B：不迁移，要求用户重新上传

第一阶段建议优先采用：

- **方案 A：自动创建默认知识库并归档旧数据**

### 10.2 删除一致性

删除知识库或文档时，需要保证三处一致：

- 关系型数据库记录删除
- LanceDB chunk 删除
- 本地原始文件删除

### 10.3 异步任务状态

文档是异步处理的，因此前端必须显示状态：

- 待处理
- 处理中
- 成功
- 失败

### 10.4 会话与知识库关系

会话引用的是“知识库 ID 列表”，不是文档列表。

原因：

- 用户认知更稳定
- 文档可持续增删，不影响会话配置结构

## 十一、验收标准

开发完成后，至少满足以下验收项：

1. 用户可以创建多个知识库
2. 知识库可以设置分类、名称、描述
3. 文档上传时必须归属某个知识库
4. 用户可以查看某个知识库中的文档列表和处理状态
5. 用户可以删除知识库和单个文档
6. 聊天会话可以引入一个或多个知识库
7. AI 回答时仅使用已引入知识库进行检索
8. AI 回答结果可以显示来源知识库和文件名
