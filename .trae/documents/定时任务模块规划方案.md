# 定时任务模块规划方案

## Summary

本次规划新增一个“定时任务模块”，用于把聊天窗口中的“定时执行意图”转化为可持久化、可确认、可审计、可调度执行的任务。第一版采用“聊天内识别意图 + LLM 总结最佳执行路径与可行性评估 + 前端弹窗二次确认 + 持久化任务 + 调度执行 + 站内通知/执行明细查看”的方案，重点解决“把一次成功完成的事情转成长期稳定运行的计划任务”。

第一版的关键定位：

- 支持用户在聊天窗口表达“加入到定时任务”“每天执行”等意图
- 结合完整会话历史、工具调用轨迹和最终结果，由 LLM 生成结构化任务草案
- 草案包含频率、时区、执行步骤、失败处理、风险、可行性验证结果
- 用户通过前端弹窗确认或修改后，才真正落库创建任务
- 调度器按用户时区执行任务，结果落到站内任务模块并保留完整执行明细
- 第一版执行工具限制为白名单工具，不直接开放所有高风险能力
- 第一版先实现站内通知与执行历史；邮件、微信保留扩展接口但不作为首批必做交付

## Current State Analysis

### 已有能力

- 聊天与会话基础链路已存在：`backend/api/v1.py`、`backend/models/chat_session.py`
- Agent 执行过程已有结构化步骤记录：`backend/models/agent_execution_step.py`、`backend/services/agent_execution_service.py`
- 前端聊天页已展示执行步骤与工具调用过程：`frontend/src/views/Chat.vue`
- Celery 基础设施已存在：`backend/workers/celery_app.py`
- 当前已有维护类定时任务样例，但没有“用户级定时任务调度模块”：`backend/workers/maintenance_tasks.py`
- 前端已有“我的任务”入口占位，但尚未接到真实任务模块：`frontend/src/views/Chat.vue`

### 当前缺口

- 没有“定时任务”领域模型、执行历史模型、通知模型
- 没有“从对话和执行轨迹生成任务草案”的 LLM 工作流
- 没有“任务确认弹窗”和“任务管理界面”
- 没有“按用户时区执行”的任务调度器
- 没有“白名单工具执行”和任务级预算/熔断/自动暂停控制
- 没有“预览执行”机制来验证任务在持久化前是否可行

### 已确认的产品决策

- 调度方式：第一版支持用户自然语言描述的时间规则，并由系统解析与归一化
- 任务固化方式：混合模式
  - LLM 总结最佳执行路径
  - 同时保留原始执行轨迹做审计
- 执行权限：白名单工具
- 确认方式：前端弹窗二次确认
- 结果投递：第一版至少支持站内通知 + 任务模块查看每次执行内容
- 扩展方向：预留邮件通知、微信通知能力
- 时区规则：按用户时区执行

## Proposed Changes

### 1. 新增调度领域模型与持久化层

#### 新增文件

- `backend/models/scheduled_task.py`
- `backend/schemas/scheduled_task.py`
- `backend/services/scheduled_task_service.py`

#### 设计内容

- 新增 `ScheduledTask`
  - `id`
  - `user_id`
  - `source_session_id`
  - `source_message_id` 或等价来源标识
  - `title`
  - `description`
  - `intent_text`
  - `schedule_type`
  - `schedule_text`
  - `cron_expression`
  - `timezone`
  - `status`：`draft / active / paused / failed / archived`
  - `delivery_channels`
  - `plan_json`
  - `raw_trace_json`
  - `feasibility_json`
  - `retry_policy_json`
  - `budget_policy_json`
  - `tool_whitelist_json`
  - `last_run_at`
  - `next_run_at`
  - `failure_count`
  - `auto_pause_reason`
  - `created_at / updated_at`

- 新增 `ScheduledTaskRun`
  - `id`
  - `scheduled_task_id`
  - `status`：`queued / running / success / failed / cancelled / skipped`
  - `triggered_at`
  - `started_at`
  - `finished_at`
  - `execution_plan_snapshot`
  - `result_summary`
  - `error_message`
  - `steps_json`
  - `notification_status_json`
  - `cost_metrics_json`

- 新增 `ScheduledTaskNotification`
  - 先支持站内通知
  - 预留 `channel / target / payload / status` 字段，为邮件和微信扩展做准备

#### 为什么这样做

- 需要把“任务定义”“执行历史”“通知投递”拆开，避免后期审计与重试逻辑混乱
- 需要保存 LLM 生成的最佳路径和原始执行轨迹，满足“可优化 + 可回看”

### 2. 新增 LLM 总结与验证工作流

#### 新增文件

- `backend/services/scheduled_task_planner_service.py`
- `backend/schemas/scheduled_task_planner.py`
- `backend/prompts/scheduled_task_planner.md` 或等价提示词文件

#### 设计内容

- 输入：
  - 会话历史
  - 最近用户意图
  - 执行步骤轨迹
  - 工具成功/失败结果
  - 当前用户上下文与知识库绑定情况

- 输出必须强制结构化，建议 Pydantic/JSON Schema，包括：
  - `title`
  - `description`
  - `schedule_text`
  - `cron_expression`
  - `timezone`
  - `delivery_channels`
  - `execution_steps[]`
  - `success_criteria[]`
  - `failure_handlers[]`
  - `preflight_checks[]`
  - `required_tools[]`
  - `required_permissions[]`
  - `estimated_cost`
  - `estimated_success_rate`
  - `risks[]`
  - `is_feasible`
  - `feasibility_reasons[]`

- 规划逻辑要求：
  - 优先提炼“最终成功”的最短稳定路径
  - 识别失败步骤并尝试给出替代策略
  - 判断缺失权限、缺失参数、缺失依赖时，直接标记不可执行

#### 为什么这样做

- 这是本需求的核心：不是简单复制历史轨迹，而是把历史对话沉淀成“可长期稳定重复执行”的计划

### 3. 新增“预览执行”和用户确认流程

#### 新增文件

- `backend/api/scheduled_tasks.py`
- `frontend/src/api/scheduledTasks.ts`
- `frontend/src/components/scheduled-task/TaskConfirmDialog.vue`

#### 设计内容

- 聊天中识别到“定时执行意图”后，不直接建任务
- 后端生成 `draft` 草案
- 前端弹出确认对话框，展示：
  - 任务名称
  - 自然语言频率 + cron 表达式
  - 用户时区
  - 最佳执行步骤
  - 成功条件
  - 风险点
  - 预估成功率
  - 需要的工具/权限
  - 通知方式

- 用户可在弹窗中：
  - 修改频率
  - 修改通知方式
  - 编辑任务名称/描述
  - 查看执行预案
  - 点“预览执行”
  - 点“确认创建”
  - 点“取消”

- “预览执行”行为：
  - 基于草案创建一次模拟运行
  - 只允许白名单工具
  - 输出预检查结果和试运行结果
  - 成功后允许确认创建

#### 为什么这样做

- 用户明确要求要有 Human-in-the-Loop
- 预览执行能显著降低错误计划直接落库的问题

### 4. 新增定时任务 API 模块

#### 新增文件

- `backend/api/scheduled_tasks.py`
- `backend/main.py` 中注册新路由

#### 计划接口

- `POST /api/scheduled-tasks/drafts`
  - 从当前会话与执行轨迹生成任务草案
- `POST /api/scheduled-tasks/drafts/{draft_id}/preview`
  - 对草案做预览执行
- `POST /api/scheduled-tasks`
  - 用户确认后正式创建任务
- `GET /api/scheduled-tasks`
  - 获取任务列表
- `GET /api/scheduled-tasks/{task_id}`
  - 获取任务详情
- `PATCH /api/scheduled-tasks/{task_id}`
  - 编辑任务
- `POST /api/scheduled-tasks/{task_id}/pause`
- `POST /api/scheduled-tasks/{task_id}/resume`
- `POST /api/scheduled-tasks/{task_id}/run-now`
- `GET /api/scheduled-tasks/{task_id}/runs`
  - 获取执行历史
- `GET /api/scheduled-tasks/runs/{run_id}`
  - 获取单次执行详情

#### 为什么这样做

- 需要把“草案”和“正式任务”分开，才能支撑确认弹窗与预览执行

### 5. 新增任务调度与执行器

#### 新增文件

- `backend/workers/scheduled_task_dispatcher.py`
- `backend/workers/scheduled_task_executor.py`
- `backend/services/scheduled_task_runtime_service.py`

#### 设计内容

- `dispatcher`
  - 定期扫描 `next_run_at <= now` 的 `active` 任务
  - 负责排队与幂等控制

- `executor`
  - 加载任务定义和快照
  - 运行 `preflight_checks`
  - 在白名单工具约束下执行计划
  - 写入 `ScheduledTaskRun`
  - 更新 `last_run_at / next_run_at / failure_count`

- 调度策略
  - 使用 Celery + 独立调度轮询任务
  - 第一版不强依赖 Celery Beat 复杂动态注册
  - 采用“数据库驱动调度 + 周期 dispatcher 扫描”的方式，便于用户级任务动态增删改

#### 为什么这样做

- 用户级任务是动态数据，不适合硬编码到静态 `beat_schedule`
- 扫描式 dispatcher 更适合复杂任务状态管理、暂停、自愈、限流与多实例扩展

### 6. 新增安全控制与自愈机制

#### 涉及文件

- `backend/services/scheduled_task_runtime_service.py`
- `backend/agents/executor.py`（按计划执行时增加工具白名单约束）

#### 设计内容

- 白名单工具执行
  - 第一版仅允许低风险工具集合
  - 禁止任务执行时自由扩展到所有 Agent 工具

- 预算与杀手开关
  - 全局开关
  - 单任务开关
  - 任务预算上限（调用次数/LLM 成本阈值）

- 自愈机制
  - 指数退避重试
  - 失败计数
  - 连续失败自动暂停
  - 高失败率时站内通知

#### 为什么这样做

- 用户明确提到“防烧钱、最小权限、自动暂停、限流、防雪崩”

### 7. 新增前端任务管理模块

#### 新增/改造文件

- `frontend/src/views/Chat.vue`
- `frontend/src/views/ScheduledTasks.vue` 或 `frontend/src/components/scheduled-task/*`
- `frontend/src/api/client.ts` 或拆出 `frontend/src/api/scheduledTasks.ts`

#### 设计内容

- 在聊天窗口：
  - 当识别出定时任务意图后，展示“生成任务草案”入口
  - 弹出确认对话框
  - 支持预览执行和确认创建

- 在“我的任务”模块：
  - 任务列表
  - 筛选状态
  - 查看详情
  - 查看每次执行内容
  - 查看风险/通知状态
  - 手动运行、暂停、恢复

#### 为什么这样做

- 现有前端已经有“我的任务”入口，占位正好可接真实模块

## Assumptions & Decisions

- 第一版的“自然语言调度”由 LLM/解析器转成标准 `cron_expression + timezone`
- 第一版以“用户时区”作为时间解释基准
- 第一版默认通知落站内；邮件/微信只做数据模型与通道扩展位，不作为首批强制交付
- 第一版采用“数据库驱动调度 + Celery dispatcher 扫描”，不做静态 `beat_schedule` 注册式方案
- 第一版必须有“draft -> preview -> confirm -> active”完整生命周期
- 第一版不做跨渠道复杂凭据托管系统，但会为通知通道预留字段
- 第一版保留原始执行轨迹做审计，但真正执行使用 LLM 总结后的最佳路径
- 第一版不直接复用所有工具，必须显式白名单

## Verification steps

1. 后端模型与 schema 验证
   - 新增任务定义、执行历史、通知模型可正常创建与查询
   - 旧系统不受影响

2. 草案生成验证
   - 从一个已有成功完成的会话生成任务草案
   - 输出包含名称、频率、cron、步骤、成功条件、失败策略、可行性评估

3. 预览执行验证
   - 对草案执行一次预览
   - 成功时返回预览结果
   - 缺权限/缺工具/缺参数时返回明确不可行原因

4. 用户确认验证
   - 前端弹窗可编辑任务草案并确认创建
   - 创建后任务状态为 `active`

5. 调度执行验证
   - dispatcher 能正确扫描到到期任务
   - executor 能在白名单约束下执行任务
   - 执行历史正确落库

6. 自愈与安全验证
   - 连续失败达到阈值后自动暂停
   - 预算限制达到阈值后停止继续执行
   - 仅白名单工具可被执行

7. 前端任务管理验证
   - “我的任务”可查看任务列表、详情、执行历史
   - 站内通知与任务执行内容可查看
