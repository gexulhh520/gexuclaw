# 文档处理优化计划

## 当前问题

1. **同步阻塞**：`_embed_and_store` 方法中串行调用 Ollama 生成 embedding，阻塞 FastAPI 事件循环
2. **大文件处理慢**：7MB PDF 可能产生几十个 chunks，处理时间长
3. **用户体验差**：处理过程中前端无反馈，不知道进度
4. **Ollama 过载风险**：并发请求可能压垮 Ollama 服务

## 优化方案

### 方案 1: 使用 Celery 后台异步处理（推荐）

将文档处理改为异步任务，上传后立即返回，后台处理完成后通知前端。

#### 实现步骤

**步骤 1: 创建文档处理 Celery 任务**

文件: `backend/workers/document_tasks.py`

@celery_app.task(bind=True)

    """
    异步处理文档任务
    
    进度状态：
    - PENDING: 等待处理
    - PROCESSING: 正在解析文档
    - EXTRACTING: 正在提取文本
    - CHUNKING: 正在分块
    - EMBEDDING: 正在生成向量 (x/total)
    - STORING: 正在存储
    - SUCCESS: 完成
    - FAILURE: 失败
    """
    try:
        # 更新进度：开始解析
        self.update_state(state='PROCESSING', meta={'progress': 10, 'message': '正在解析文档...'})
        
        # 发送 WebSocket 通知

            connection_manager.send_to_session(session_id, {
                'type': 'document_progress',
                'status': 'processing',
                'progress': 10,
                'message': '正在解析文档...'
            })
        
        # 处理文档...
        # （每个阶段更新进度）

### 方案 2: 限制并发 + 前端进度（备选）

如果不使用 Celery，可以：

1. **限制并发**：使用信号量限制同时处理的文档数
2. **分批次处理**：每处理几个 chunks 让出事件循环
3. **WebSocket 推送进度**：实时通知前端处理进度

## 推荐方案

**推荐方案 1（Celery 异步处理）**：

优点：
- 完全不阻塞主服务
- 支持任务重试
- 可扩展性强
- 用户上传后立即得到响应

缺点：
- 需要额外的 Celery Worker
- 实现复杂度稍高

## 实施计划

### 阶段 1: 基础异步处理
1. 创建 `document_tasks.py` Celery 任务
2. 修改 `upload.py` 接口提交异步任务
3. 前端轮询任务状态

### 阶段 2: 进度通知（可选增强）
1. 任务中发送 WebSocket 进度通知
2. 前端实时显示进度条

### 阶段 3: 并发控制（可选增强）
1. 限制同时处理的文档数量
2. 队列管理，避免 Ollama 过载

## 验收标准

1. 上传大文档（7MB+）不阻塞其他接口
2. 上传后立即返回响应（< 1秒）
3. 前端能显示处理进度或状态
4. 处理完成后能正常使用知识库搜索
