# Vue 3 + TypeScript + Vite 前端提示词模板

这份模板适合你后续在现有项目里直接复制使用。  
目标不是“让模型随便写个能跑的页面”，而是尽量让它输出：

- 视觉统一
- 交互完整
- 代码可维护
- 符合现有项目风格
- 能接着你当前项目继续开发

---

## 1. 项目总提示词

适合在你给模型提需求前，先贴这一段作为“总规则”。

```text
你是资深前端工程师和 UI 落地工程师，请基于我当前的 Vue 3 + TypeScript + Vite 项目继续开发。

核心目标不是“能用”，而是“好看、好用、可维护、符合现有项目风格”。

项目要求：
- 技术栈：Vue 3 + TypeScript + Vite
- 尽量沿用当前项目已有的目录结构、代码风格、组件封装方式和命名习惯
- 不要为了完成任务额外引入不必要的新库
- 如果项目里已经有现成组件、工具函数、样式变量、hooks、store、接口封装，请优先复用
- 如果需要新增抽象，必须保持轻量，避免过度设计

视觉要求：
- 页面要有明确层级和视觉焦点
- 留白、间距、字号、圆角、阴影、边框要统一
- 不要输出默认模板感很强的页面
- 不要使用廉价、杂乱、俗套的配色
- 设计风格要和当前项目保持一致
- 页面要有完成度，像真实产品而不是 demo

交互要求：
- 补全 loading / empty / error / disabled 状态
- 补全 hover / active / focus 反馈
- 表单校验、按钮反馈、空状态、提示信息都要考虑
- 交互尽量自然克制，不要花哨
- 移动端和桌面端都要尽量可用

代码要求：
- 使用 Composition API
- 使用 TypeScript 并尽量补全类型
- 组件拆分合理，避免单文件过长
- 命名语义化，避免无意义缩写
- 优先复用现有项目能力，不要重复造轮子
- 保持代码整洁，可读，可扩展

输出要求：
- 先简述实现思路
- 再给出完整代码
- 如果需要拆文件，请按文件路径分别输出
- 如果有可能更好的实现方式，请最后单独说明

在开始编码前，请先快速判断：
1. 这个需求在当前项目里应该放在哪个目录
2. 是否已有可复用的组件或工具
3. 需要补哪些状态和交互
4. 需要遵守哪些现有约束
如果信息不够，请先基于代码库做合理判断，不要直接臆造。
```

## 2. 页面生成模板

适合新增页面、详情页、列表页、表单页、仪表盘等。

```text
请基于我当前的 Vue 3 + TypeScript + Vite 项目实现一个页面。

页面名称：{页面名称}
页面用途：{这个页面解决什么问题}
所在模块：{例如 用户管理 / 订单管理 / 内容管理 / 工作台}

请优先参考当前项目已有页面的结构、布局、组件风格和样式规范，保持一致性。

页面目标：
- 视觉上要有设计感，不能像默认后台模板
- 信息层级清晰，主次分明
- 交互完整，状态齐全
- 能和现有项目无缝融合

页面结构：
- 顶部区域：{标题 / 描述 / 操作按钮}
- 主体区域：{表格 / 卡片 / 表单 / 图表 / 列表}
- 辅助区域：{筛选区 / 侧边栏 / 详情面板 / 弹窗}
- 底部区域：{分页 / 说明 / 汇总}

视觉要求：
- 使用统一的间距和排版节奏
- 主按钮、次按钮、文本、提示的层级要清楚
- 背景不要太单调，可以适当用渐变、层次背景、分割线、卡片阴影增强质感
- 不要让所有元素平均用力

交互要求：
- 支持 loading、empty、error、disabled 状态
- hover、focus、active 要有明确反馈
- 如果有筛选、搜索、分页、弹窗、抽屉，交互要完整
- 任何会等待接口的地方都要有反馈

工程要求：
- 使用 Vue 3 Composition API
- 使用 TypeScript
- 组件拆分合理
- 如果涉及接口调用，请把数据层和视图层分开
- 尽量复用项目现有组件和样式变量

请输出：
1. 页面实现思路
2. 组件拆分建议
3. 完整代码
4. 如果有需要接入现有项目的地方，请明确说明
```

## 3. 组件生成模板

适合按钮组、筛选条、卡片、空状态、标签、状态徽标、表单项等可复用组件。

```text
请基于我当前的 Vue 3 + TypeScript + Vite 项目实现一个可复用组件。

组件名称：{组件名}
使用场景：{在哪些页面用}
组件职责：{这个组件负责什么}

请优先沿用当前项目已有的组件风格、props 命名方式和样式规范。

组件设计要求：
- 接口清晰，props / emits / slots 定义明确
- 组件要足够通用，但不要过度抽象
- 默认态、hover、active、disabled、loading 状态齐全
- 视觉上要精致，避免廉价感
- 如果是业务组件，要考虑真实数据和边界情况

工程要求：
- 使用 Vue 3 Composition API
- 使用 TypeScript 并补全类型
- 组件内部逻辑清晰，不要把业务写死
- 如果有示例，提供一个简单使用示例

请输出：
1. 组件设计说明
2. Vue 单文件组件代码
3. 示例用法
4. 如果你认为还需要配套的辅助文件，也一起给出
```

## 4. 表单页面模板

适合新增、编辑、审核、配置类页面。

```text
请基于我当前的 Vue 3 + TypeScript + Vite 项目实现一个表单页面。

表单名称：{表单名称}
业务场景：{新增 / 编辑 / 审核 / 配置}
字段内容：{列出字段}

请保证这个表单页面既实用又有质感，不要写成普通的字段堆叠页面。

要求：
- 布局清晰，字段分组合理
- 重点字段有视觉强调
- 表单提示、校验、错误反馈完整
- 保存、取消、重置、提交等操作反馈明确
- 如果字段较多，要考虑分区、折叠、锚点或分步设计
- 兼容桌面端和移动端

实现要求：
- 使用 Vue 3 Composition API
- 使用 TypeScript
- 尽量复用项目现有表单组件
- 如果涉及接口，区分初始化数据、提交数据、错误处理
- 保持代码清晰，方便后续扩展字段

请输出：
1. 表单结构设计
2. 验证和交互说明
3. 完整代码
```

## 5. 列表页模板

适合表格、筛选、分页、批量操作页面。

```text
请基于我当前的 Vue 3 + TypeScript + Vite 项目实现一个列表页。

列表名称：{列表名称}
业务场景：{用户 / 订单 / 内容 / 商品 / 任务 等}

页面要求：
- 筛选区清晰，搜索逻辑合理
- 表格信息密度适中
- 空状态、加载状态、错误状态完整
- 支持分页、排序、刷新、批量操作
- 操作区层级清楚，不要拥挤

视觉要求：
- 列表页要有节奏感，不要只是表格直接铺满
- 工具栏、筛选区、表格区、分页区之间要有清晰分隔
- 主操作和次操作层级分明

实现要求：
- 使用 Vue 3 Composition API
- 使用 TypeScript
- 复用当前项目里已有的表格、分页、筛选组件
- 数据请求和页面展示要分层
- 如果有权限控制，请预留处理方式

请输出：
1. 页面布局方案
2. 数据交互说明
3. 完整代码
```

```text

期望结果：{期望}
实际结果：{实际}

请先阅读相关代码，结合项目现有写法定位问题，不要直接凭空改。

要求：
- 先分析根因

- 尽量不影响其他页面和组件

请输出：
1. 问题原因分析
2. 修改方案
3. 具体代码改动
4. 需要验证的点
```

## 7. 重构优化模板

```text
重构目标：
- 提升可读性

- 降低耦合
- 保持行为不变

请注意：
- 不要为了重构而过度抽象
- 不要破坏现有功能
- 不要引入不必要的新依赖
- 优先沿用项目原有架构

我希望你：
1. 先指出这段代码的问题
2. 再说明你准备怎么改
3. 最后给出重构后的代码

如果你认为某些改动风险较高，请提前说明。
```

适合让模型检查你写的前端代码质量。

```text
请以资深前端代码审查的标准检查我当前的 Vue 3 + TypeScript + Vite 代码。

审查重点：
- 可读性
- 组件拆分是否合理
- TypeScript 类型是否完整
- 状态处理是否完善
- 交互体验是否有遗漏
- 是否符合现有项目风格
- 是否存在潜在 Bug 或维护风险

请优先指出：
- 明显问题
- 高风险问题
- 体验缺失
- 可以立刻优化的点

输出格式：
1. 问题清单
2. 风险等级
3. 建议修改方案
4. 如果需要，我再让你直接改代码
```

1. 先贴 `项目总提示词`

3. 最后补上具体需求、字段、接口、截图或设计参考
